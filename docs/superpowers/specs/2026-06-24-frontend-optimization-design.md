# 前端优化设计（2026-06-24）

## 背景与目标

现有前端是单个 `static/index.html`（约 900 行，原生 JS、无构建链），由 FastAPI 同源托管，面向**前端开发者**，承载两个场景：`code-qa`（代码问答）与 `troubleshoot`（问题排查）。本次从真实用户体验出发做一轮优化。

经过摩擦点梳理，确定本次解决五类问题（按摩擦点编号）：

- **A 场景切换不顺 + 顶栏留白**
- **B 等待反馈太弱**
- **C 消息级操作缺失**
- **D 不响应式**
- **F 难扩展（单文件 900 行）**

> E（历史搜索/重命名/分组）本次不做。

同时整体**视觉风格切换为「马卡龙软萌风」**：以项目根目录的 `proto.html` 为视觉蓝本，全面采纳其配色、玻璃拟态、大圆角、回弹动画、emoji 头像/场景图标、云朵 toast、回到底部悬浮按钮——但**文案保持专业中性口径**（标题、按钮、提示沿用现有措辞，不用「软萌」「可爱对话」等口吻）。详见第 4 部分与附录「设计令牌」。

**三条总原则（已与用户确认）：**

1. **0 后端改动**——不动 `server.py` / SSE 协议 / `/scenarios` 字段。
2. **0 构建链**——继续原生 JS + ES 模块 + CDN（marked / highlight.js），不引入框架或打包工具。
3. **视觉照 `proto.html`、文案保持专业**——采纳整套视觉系统，不采纳其可爱文案。

---

## 第 1 部分 · 架构地基：静态模块拆分（修复 F）

把单文件拆成职责清晰的多文件，**先做纯搬迁、不夹带功能改动**，让后续 A/B/C/D 各自落到对应模块。

```
static/
├── index.html          # 仅结构骨架 + <link>/<script type="module">
├── styles/
│   └── app.css         # 现有 <style> 全量搬出（第 1 步主题不变，第 4 部分再换皮）
└── js/
    ├── main.js         # 入口：init、hash 路由、串起各模块
    ├── api.js          # 后端对接：/scenarios、/ask_stream(SSE 解析)、/conversations
    ├── sessions.js     # localStorage 历史的增删改查
    ├── scenarios.js    # 场景元数据：图标分配、guide
    ├── render.js       # Markdown 渲染、代码高亮、复制按钮、气泡 DOM
    └── ui.js           # 顶栏/侧栏/输入框/toast 等界面状态
```

**要点**

- 模块间用 ES `import/export`，`<script type="module">` 直接加载，零构建、零新增依赖。
- 拆分后，A/B/C/D 改动各自收敛在对应文件，不再翻整个大文件。
- 验收：拆分完成后页面行为与现状**完全一致**（纯重构，无可见变化）。

## 第 2 部分 · 布局改造（A + D）

### 2.1 顶栏场景切换器（A）

- 顶栏的场景徽章升级为**可点下拉**：点击展开场景列表，选择某场景。
- 后端约束：会话首轮即锁定场景，无法中途切换。因此选择**不同**场景的语义是「**在该场景下开一条新对话**」，省掉「先点新对话」的绕路。
- 若当前对话已进行（已有消息），切换到其它场景前给一条轻量确认（toast 或就地确认），避免误丢当前上下文。
- 选择**同一**场景不产生新对话。

### 2.2 顶栏右侧信息（A）

- 顶栏右侧不再留白，常驻显示**本会话累计轮数 + 花费**（`num_turns` / `cost_usd` 数据已有，逐轮累加展示）。

### 2.3 侧栏可折叠（D）

- 顶栏左侧加 ☰ 折叠按钮，一键收起/展开侧栏，主区相应变宽。
- 折叠状态写入 localStorage，刷新后保持。

### 2.4 窄屏降级（D，目标：窄屏桌面 / 分屏）

- 窗口宽度 < ~760px 时，侧栏自动收为**浮层**：默认隐藏，点 ☰ 临时滑出，点击主区或选中会话后自动收起。
- 消息流与输入框占满宽度；与 IDE 并排 / 分屏时不被挤坏。
- **不做**完整移动端（触控手势、虚拟键盘适配等）。

## 第 3 部分 · 对话体验（B + C，纯前端）

### 3.1 等待反馈（B）

- **停止按钮**：回答流式期间，发送按钮 ✈ 变为停止键 ■；点击中断本次 SSE（abort fetch / 关闭流），**已流出的文字保留**，输入框恢复可用。
- **分级安抚文案**（按已用时长切换，缓解「像卡死」）：
  - 0–10s：`正在思考`
  - 10–30s：`正在查阅相关代码 / 日志`
  - 30s+：`仍在分析，复杂排查可能需要 1–2 分钟…`
- 保留现有跳动圆点动画与**秒级计时**。
- 这是按时间猜测的文案，非真实阶段；**真实阶段**需后端加 SSE progress 事件，本次**不做**。

### 3.2 消息级操作（C）

- **助手气泡悬停浮出操作条**（不抢视觉）：
  - `复制整条`：复制该条回答的原始 Markdown 文本。
  - `重新生成`：用产生该回答的同一问题再发一次。
- 代码块仍保留各自的一键复制按钮。
- **出错气泡加「一键重试」**：用刚才那条问题重新发送（当前实现缺此按钮，spec 曾要求）。

## 第 4 部分 · 视觉换皮（马卡龙软萌风）+ 场景图标（F）

整体视觉以 `proto.html` 为蓝本换皮；**文案保持专业中性**。

### 4.1 场景图标 / 头像：emoji，前端随机分配（F）

- 删除现有 `scnIcon()` 的正则名字猜测，改用 **emoji 图标**（贴合 proto.html 风格，如 💻 🔧 📄 🗄️ 🐛 ⚡ 等）。
- 前端**稳定随机**：按场景名做哈希取模，从内置 emoji 集中分配一个——同一场景每次渲染一致、不闪烁，新增场景无需写规则。
- 头像同样改为 proto 的**渐变圆 + emoji**：助手用薄荷绿渐变 🤖、用户用粉色渐变（或「你」字）。
- **不改后端**（`/scenarios` 不新增 `icon` 字段），**不做**示例问题（`examples`）。

### 4.2 马卡龙视觉系统（照 `proto.html` 采纳）

- **配色**：粉 / 蓝 / 薄荷 / 薰衣草 / 奶油黄马卡龙色系；页面背景柔和渐变 `linear-gradient(145deg,#fef7fb,#f0f7ff)`。令牌见附录。
- **玻璃拟态**：侧栏、顶栏、输入区用半透明白 + `backdrop-filter: blur()`。
- **大圆角 + 柔和阴影**：气泡 / 卡片 / 按钮采用 proto 的 `--r-xl…--r-sm` 与蓝/粉柔和投影。
- **气泡**：助手玻璃白底、用户粉色渐变；尾角小圆角（6px）。
- **回弹动画**：`cubic-bezier(.34,1.56,.64,1)`；消息 `popIn` 弹入、欢迎头像 `float` 浮动、思考圆点 `cuteBlink`、按钮 hover 微缩放。
- **云朵 toast**：底部居中、渐变色，分 success / error / info 三态，替换现有顶部 toast。
- **回到底部悬浮按钮**：消息流上滚离底时浮现（proto 已有该交互），一键回到最新。
- **文案红线**：标题仍为「后端问答AI助手」，按钮仍为「新对话」，等待文案沿用第 3 部分的分级专业文案——**不采纳** proto 的「软萌」「可爱对话」「正在认真思考」等措辞。
- 不引入新框架；以 CSS 变量集中管理令牌，便于后续调色。

## 不做（YAGNI / 明确排除）

- E：历史搜索 / 重命名 / 按场景分组 / 跨设备同步。
- 后端 SSE progress「真实排查阶段」事件。
- 场景示例问题（`examples`）。
- 后端 `/scenarios` 新增 `icon` 字段。
- 完整移动端适配（仅做到窄屏桌面 / 分屏）。
- 任何构建链 / 前端框架。

## 对接点（均为现有接口，无需改造）

| 用途 | 接口 |
| --- | --- |
| 列出场景 | `GET /scenarios` |
| 流式提问 / 追问 | `POST /ask_stream`（SSE：`token` / `done` / `error`） |
| 加载某会话历史 | `GET /conversations/{id}/messages` |

## 验收要点

1. 第 1 部分拆分后，页面行为与现状一致（纯重构）。
2. 顶栏可切换场景（切到异场景=新对话）、显示累计轮数与花费。
3. 侧栏可折叠并记忆；窗口 < ~760px 时侧栏转浮层、主区占满。
4. 流式期间可停止并保留已出文字；等待文案随时长分级；计时正常。
5. 助手气泡可复制整条 / 重新生成；出错气泡可一键重试。
6. 场景图标/头像为 emoji、按名稳定分配，新增场景无需改前端规则。
7. 视觉换皮到马卡龙风（配色/玻璃拟态/圆角/回弹动画/云朵 toast/回到底部按钮）与 `proto.html` 一致，但文案保持专业中性。
8. 全程无后端改动、无构建链。
